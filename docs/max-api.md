# Работа с MAX API

Бот общается с MAX через Bot API. В проекте используется Python-библиотека `maxapi`, установленная как обычная pip-зависимость:

```bash
python -m pip install "maxapi[fastapi]>=1.1,<2"
```

В коде прямой импорт из локальной папки `maxapi` не используется. Адаптер находится в `app/bot/client.py`: он создаёт `maxapi.Bot`, отправляет сообщения, редактирует сообщения, удаляет сообщения и приводит inline-клавиатуры к формату, который ожидает библиотека.

## Зачем нужен webhook

Webhook — это публичный HTTP-адрес, на который MAX сам присылает события бота. Пользователь нажал `/start`, отправил текст или нажал кнопку — MAX делает `POST` на наш endpoint.

Для этого проекта выбран webhook, а не long polling, потому что:

- Cloud Functions просыпается только на входящее событие и не требует постоянно работающего процесса;
- не нужен сервер, который держит бесконечный цикл опроса;
- проще остаться в бесплатной serverless-архитектуре;
- MAX быстрее доставляет события, а функция быстро отвечает `200`.

Long polling удобен в локальной разработке, но для Cloud Functions он плохо подходит: функция не должна бесконечно ждать события, она должна отработать короткий запрос и завершить обработку.

## Публичный URL webhook

В Yandex Cloud Functions прямой URL выглядит так:

```text
https://functions.yandexcloud.net/<function_id>
```

Для текущего стенда:

```text
https://functions.yandexcloud.net/d4en27dqbb87rvb7lmms
```

В локальном FastAPI путь webhook:

```text
http://localhost:8080/webhook
```

Но MAX не сможет сам достучаться до `localhost` на вашей машине. Локальный URL подходит только для тестов через `scripts/send-sample-update.*`, Postman или временный tunnel-сервис. Для реальной подписки MAX нужен публичный HTTPS URL.

## Секрет webhook

Переменная:

```env
WEBHOOK_SECRET=...
```

нужна, чтобы отличать реальные события MAX от чужих POST-запросов. При регистрации подписки в MAX указывается тот же secret. Когда MAX вызывает webhook, он передаёт заголовок:

```http
X-Max-Bot-Api-Secret: <secret>
```

Код нормализует заголовки к нижнему регистру и сравнивает значение с `WEBHOOK_SECRET`. Если secret неверный или отсутствует, функция отвечает:

```json
{"error":"Forbidden"}
```

со статусом `403`.

Нюансы:

- secret не должен совпадать с токеном бота;
- secret нельзя коммитить;
- если поменяли secret в Cloud Functions, надо поменять его и в подписке MAX;
- если `WEBHOOK_SECRET` пустой, проверка отключается, но для продакшена так делать не надо.

Сгенерировать secret можно так:

```bash
python -c "import secrets; print(secrets.token_hex(32))"
```

## Токен бота

Переменная:

```env
MAX_BOT_TOKEN=...
```

используется для исходящих запросов к MAX API: отправить сообщение, отредактировать сообщение, удалить сообщение, отправить уведомление.

Токен берётся в MAX у инструмента создания/настройки бота. Это секрет. Его нельзя показывать в документации, логах, скриншотах и публичных репозиториях.

## Какие update-события обрабатываются

Входящие события разбирает `app/bot/dispatcher.py`.

Поддерживаются:

| `update_type` | Что означает | Как используется |
| --- | --- | --- |
| `bot_started` | Пользователь впервые запустил бота, нажал стартовое действие или перешёл по диплинку. | Создаём/обновляем пользователя и показываем дисклеймер, каталог или конкретное мероприятие из `payload`. |
| `message_created` | Пользователь отправил обычное сообщение. | Обрабатываем `/start`, `/events`, `/my`, `/organizer`, `/find КОД` и текстовые команды. |
| `message_callback` | Пользователь нажал inline-кнопку. | Распаковываем payload и открываем карточку, создаём запись, отменяем запись, переключаем уведомления и так далее. |

Если тип события неизвестен, обработчик его игнорирует и всё равно возвращает `200` на уровне webhook. Это нормально: MAX может добавлять события, которые боту не нужны.

## Формат локального тестового webhook

Пример `/start`:

```json
{
  "update_type": "message_created",
  "message": {
    "sender": {
      "user_id": 101,
      "name": "Локальный пользователь"
    },
    "recipient": {
      "chat_id": 9001
    },
    "body": {
      "text": "/start",
      "mid": "local-user-message-1"
    }
  }
}
```

PowerShell:

```powershell
$headers = @{
  "Content-Type" = "application/json"
  "X-Max-Bot-Api-Secret" = "change_me_secret"
}

$body = @{
  update_type = "message_created"
  message = @{
    sender = @{ user_id = 101; name = "Локальный пользователь" }
    recipient = @{ chat_id = 9001 }
    body = @{ text = "/start"; mid = "local-user-message-1" }
  }
} | ConvertTo-Json -Depth 10

Invoke-RestMethod `
  -Method Post `
  -Uri "http://localhost:8080/webhook" `
  -Headers $headers `
  -Body $body
```

Linux/macOS:

```bash
curl -sS -X POST 'http://localhost:8080/webhook' \
  -H 'Content-Type: application/json' \
  -H 'X-Max-Bot-Api-Secret: change_me_secret' \
  -d '{
    "update_type": "message_created",
    "message": {
      "sender": {"user_id": 101, "name": "Локальный пользователь"},
      "recipient": {"chat_id": 9001},
      "body": {"text": "/start", "mid": "local-user-message-1"}
    }
  }'
```

В проекте уже есть скрипты:

```powershell
.\scripts\send-sample-update.ps1
```

```bash
bash scripts/send-sample-update.sh
```

Они отправляют `bot_started` на локальный webhook.

## Ссылки на конкретные мероприятия

MAX передаёт контекст диплинка через `bot_started.payload`. Для мероприятий проект использует формат:

```text
https://max.ru/<botName>?start=e_<event-slug>
```

Пример:

```text
https://max.ru/id123_bot?start=e_it-open-day-2026-06-15
```

Ограничения:

- `payload` MAX не должен быть длиннее 128 символов;
- бот принимает только `e_<slug>`;
- `slug` состоит из латинских строчных букв, цифр и дефиса;
- если ссылка устарела или некорректна, webhook всё равно отвечает `200`, а пользователь видит мягкую ошибку и обычный вход в бота;
- для старых мероприятий без slug бот создаёт slug при первом показе карточки;
- для показа ссылок в карточках бот использует `MAX_BOT_USERNAME`, а если переменная пустая, пробует получить ник через `GET /me`;
- карточка не печатает raw URL в тексте: она показывает `Ссылка: Нажмите чтобы скопировать` и кнопку `Поделиться` типа `clipboard`;
- в подписке MAX должен быть включён update-тип `bot_started`.

Пример входящего webhook от MAX:

```json
{
  "update_type": "bot_started",
  "chat_id": 9001,
  "user": {
    "user_id": 101,
    "name": "Анна"
  },
  "payload": "e_it-open-day-2026-06-15"
}
```

## Пример callback-события

Inline-кнопки отправляют payload. В проекте payload — это компактная строка:

```text
action|event_id|slot_id|registration_id|value
```

Например:

```text
event_detail|123
```

или:

```text
register_confirm|123||456
```

Поля справа могут отсутствовать. Упаковка и распаковка находятся в `app/bot/payloads.py`.

Пример callback webhook:

```json
{
  "update_type": "message_callback",
  "callback": {
    "user": {
      "user_id": 101,
      "name": "Локальный пользователь"
    },
    "payload": "event_detail|123"
  },
  "message": {
    "recipient": {
      "chat_id": 9001
    },
    "body": {
      "mid": "bot-message-id-from-max"
    }
  }
}
```

`message.body.mid` здесь важен: если он есть, бот пытается отредактировать это сообщение вместо отправки нового.

## Inline-кнопки

Кнопки создаются в `app/bot/keyboards.py`.

Формат кнопки:

```json
{
  "type": "callback",
  "text": "1 ℹ️ День открытых дверей...",
  "payload": "event_detail|123",
  "intent": "default"
}
```

Для копирования ссылки на мероприятие используется кнопка MAX типа `clipboard`:

```json
{
  "type": "clipboard",
  "text": "🔗 Поделиться",
  "payload": "https://max.ru/id123_bot?start=e_it-open-day-2026-06-15"
}
```

Клавиатура:

```json
[
  {
    "type": "inline_keyboard",
    "payload": {
      "buttons": [
        [
          {
            "type": "callback",
            "text": "✅ Согласен",
            "payload": "consent_accept",
            "intent": "default"
          }
        ]
      ]
    }
  }
]
```

В `maxapi` вложения ожидают объект с методом `model_dump`. Поэтому адаптер `_RawAttachment` оборачивает обычные `dict`, чтобы библиотека смогла сериализовать клавиатуру.

## Отправка сообщений

Исходящая отправка идёт через:

```python
MaxApiBotClient.send_message(...)
```

Он вызывает:

```python
self.bot.send_message(...)
```

и возвращает `message_id`, если MAX API вернул его в ответе. Этот id нужен для последующего удаления или редактирования.

Сообщение может отправляться по `user_id` или по `chat_id`. В личном диалоге с ботом обычно достаточно `user_id`; для событий из webhook проект также передаёт `chat_id`, если он пришёл в update.

## Редактирование сообщений

Если пользователь нажимает inline-кнопку, MAX присылает callback с id сообщения бота. Тогда обработчик пытается:

```python
edit_message(message_id=source_message_id, ...)
```

Это делает интерфейс чище: карточка мероприятия открывается на месте каталога, подтверждение записи открывается на месте карточки, и чат не превращается в длинную ленту меню.

Если редактирование не удалось, бот отправит новое сообщение. Это сделано как запасной путь: лучше прислать пользователю ответ новым сообщением, чем молча сломать сценарий.

## Удаление предыдущих сообщений

Для обычных текстовых команд вроде `/start` у нас нет id сообщения бота в текущем update. Поэтому проект хранит последнее отправленное сообщение бота в таблице YDB:

```text
bot_sessions
```

Поле:

```text
last_bot_message_id
```

При новой отправке бот:

1. читает предыдущий `last_bot_message_id`;
2. пытается удалить это сообщение через MAX API;
3. отправляет новое сообщение;
4. сохраняет новый id.

Что важно:

- сообщения, отправленные до появления этого механизма, не были сохранены и могут остаться;
- MAX API может запретить удаление некоторых сообщений;
- если удаление упало, бот всё равно отправит новый ответ, чтобы пользователь не остался без реакции;
- callback-сценарии чаще редактируют сообщение, а текстовые команды чаще заменяют последнее сообщение.

## Регистрация webhook в MAX

Подписка создаётся через метод MAX API `POST /subscriptions`. Точный внешний вид запроса лучше сверять с актуальной официальной документацией MAX, но смысл такой:

- указывается публичный URL функции;
- указывается набор update-событий;
- указывается secret, совпадающий с `WEBHOOK_SECRET`.

Шаблон curl:

```bash
curl -sS -X POST 'https://botapi.max.ru/subscriptions' \
  -H 'Authorization: Bearer <MAX_BOT_TOKEN>' \
  -H 'Content-Type: application/json' \
  -d '{
    "url": "https://functions.yandexcloud.net/<function_id>",
    "update_types": [
      "bot_started",
      "message_created",
      "message_callback"
    ],
    "secret": "<WEBHOOK_SECRET>"
  }'
```

Если в официальной документации MAX для токена используется query-параметр или другой заголовок, используйте официальный вариант. Важно не название поля в примере, а три обязательные идеи: URL, события, secret.

Текущий публичный URL стенда:

```text
https://functions.yandexcloud.net/d4en27dqbb87rvb7lmms
```

## Производительность и лимиты

Webhook должен отвечать быстро. MAX ожидает HTTP-ответ, и если функция долго думает или падает, пользователь видит задержки. Поэтому тяжёлая логика не должна выполняться в webhook синхронно.

В проекте короткие действия выполняются сразу:

- показать каталог;
- открыть карточку;
- создать запись;
- отменить запись;
- показать мои записи.

Массовые уведомления идут через таблицу:

```text
notification_outbox
```

Timer trigger вызывает функцию раз в несколько минут, а `NotificationWorker` выбирает due-уведомления и отправляет их с ограничением:

```env
MAX_API_RPS=30
```

`RPS` — requests per second, то есть запросов в секунду. Значение `30` означает, что worker не должен отправлять больше 30 сообщений в секунду. Если MAX API опубликует другой лимит для вашего тарифа/бота, значение надо снизить.

## Почему был важен `Event loop is closed`

`maxapi` внутри использует асинхронную HTTP-сессию. Если создать клиента один раз, а потом каждый webhook запускать через `asyncio.run`, Python после каждого вызова закрывает event loop. Клиент остаётся жить, но его внутренняя сессия привязана к уже закрытому loop. Следующий запрос падает с:

```text
RuntimeError: Event loop is closed
```

В проекте это решено через постоянный `_AsyncRunner` в `app/function_handler.py`. Он держит один event loop на инстанс Cloud Function и переиспользует его между вызовами. Поэтому MAX-клиент не оказывается привязан к закрытому loop.

Если эта ошибка вернулась, проверьте:

- не создаётся ли новый `asyncio.run` вокруг старого клиента;
- не был ли изменён `function_handler.py`;
- не создаётся ли `MaxApiBotClient` в одном loop, а используется в другом.

## Частые ошибки

`403 Forbidden` от webhook.

Почти всегда неверный `WEBHOOK_SECRET`. Проверьте secret в Cloud Functions и secret в подписке MAX. Они должны совпадать.

MAX не вызывает webhook.

Проверьте, что URL публичный и начинается с `https://`. `localhost`, `127.0.0.1` и адреса внутри Docker-сети MAX не увидит.

Функция отвечает `502`.

Смотрите логи Cloud Functions. Частые причины: ошибка импорта зависимости, неверный `YDB_ENDPOINT`, нет прав у service account, сломан event loop.

Кнопка нажимается, но ничего не меняется.

Проверьте, приходит ли `message_callback` в логи. Если приходит и есть ошибка, смотреть stack trace. Если не приходит, проблема в подписке MAX или payload кнопки.

Старые сообщения не удаляются.

Скорее всего, они были отправлены до сохранения `last_bot_message_id`, или MAX API не разрешил удалить конкретное сообщение. После новой версии бот начинает чистить новые свои сообщения.

Сообщение не помещается в интерфейс MAX.

Кнопки должны быть короткими. В каталоге показываем только номер, иконку и короткое название. Полная информация открывается в карточке мероприятия.

Уведомления не отправляются.

Проверьте `notification_outbox`, timer trigger, переменную `MAX_API_RPS` и логи вызовов таймера. Timer-событие отличается от HTTP webhook: оно приходит в поле `messages` с типом `yandex.cloud.events.serverless.triggers.TimerMessage`.
