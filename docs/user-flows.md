# Пользовательские сценарии

Этот документ описывает, как бот ведет разные роли. Он не заменяет код, но помогает быстро понять, какие сценарии считаются основными и где искать их реализацию.

Термины пользовательского интерфейса сверяются с [глоссарием](glossary.md): говорим `мероприятие`, `Организатор`, `Записан`.

## Роли

В проекте есть три пользовательские роли.

Обычный пользователь — абитуриент или другой участник, который записывается на мероприятия.

Организатор — пользователь, который управляет мероприятиями, видит участников и отмечает посещение.

Администратор — пользователь, который назначает и снимает роль Организатора.

Технически роли хранятся в таблице `role_assignments`. Роли из `ADMIN_USER_IDS` и `ORGANIZER_USER_IDS` синхронизируются при старте приложения или создании Cloud Function handler.

## Первый вход

Пользователь попадает в бота через `/start`, стартовую кнопку MAX или диплинк на конкретное мероприятие.

Если у пользователя еще нет согласия, бот показывает дисклеймер. В нем прямо сказано, что сервис хранит только минимальные данные профиля MAX, выбранное мероприятие и статус записи. После согласия создается строка в `consents`.

Если пользователь пришел по ссылке на мероприятие, после согласия бот открывает карточку этого мероприятия. Если ссылка неверная или мероприятие больше недоступно, бот показывает мягкую ошибку и возвращает пользователя к обычному входу.

Код: `BotHandlers.handle_bot_started`, `_send_entrypoint`, `_send_deeplink_entrypoint`.

## Главное меню

Главное меню дает быстрый доступ к каталогу, моим записям и, если есть роль, к меню Организатора или администратора.

Команды:

```text
/start или /menu
/events
/my или /records
/organizer
/admin
/find КОД
```

Команда `/find КОД` доступна Организаторам и ищет запись по коду среди доступных им мероприятий.

## Каталог мероприятий

Каталог показывает будущие мероприятия. Прошедшие мероприятия обычному пользователю не показываются. Если мероприятие разделено на слоты, свободные места считаются по слотам; если слотов нет — по общей вместимости.

Карточка мероприятия показывает:

- название;
- описание;
- дату и время;
- длительность;
- формат: онлайн или очно;
- требования;
- адрес или ссылку;
- количество свободных мест;
- кнопку записи;
- кнопку копирования публичной ссылки, если доступен `MAX_BOT_USERNAME`.

Диплинк строится как:

```text
https://max.ru/<botName>?start=e_<event-slug>
```

Slug хранится в `event_deeplinks`. Для старых мероприятий без slug бот может создать его при первом показе карточки.

## Запись на мероприятие

Основной сценарий:

1. Пользователь открывает карточку мероприятия.
2. Нажимает кнопку записи.
3. Если есть слоты, выбирает слот.
4. Видит краткое подтверждение.
5. Подтверждает запись.
6. Получает код записи.

Создание записи выполняет `RegistrationService.create_registration`, а фактическую атомарную операцию делает хранилище.

Важные правила:

- без согласия записываться нельзя;
- на одно мероприятие нельзя иметь две активные записи;
- нельзя записаться, если регистрация закрыта;
- нельзя записаться на прошедшее мероприятие;
- нельзя записаться сверх вместимости;
- код записи должен быть уникальным.

В YDB это защищается транзакцией и служебными таблицами `registration_codes` и `active_registration_keys`.

## Мои записи

Раздел “Мои записи” показывает записи пользователя. Из него можно:

- открыть карточку записи;
- отменить запись;
- включить или отключить уведомления.

Если пользователь отменяет запись до начала мероприятия, место возвращается в доступную вместимость. Поздняя отмена зависит от политики мероприятия: она может быть запрещена или разрешена с отдельным статусом.

Статус активной записи в пользовательском UI называется `Записан`.

## Уведомления пользователю

Автоматические напоминания создаются при записи и попадают в `notification_outbox`. Обычно это напоминания за сутки и за час, если конкретный текст формируется текущими правилами `app/services/reminders.py`.

Пользователь может отключить уведомления по конкретной записи. Тогда worker при отправке увидит, что уведомления отключены, и пометит задачу как `skipped`.

Ручные уведомления Организатора тоже идут через `notification_outbox`, а не отправляются прямо из callback-обработчика.

## Меню Организатора

Организатор видит список мероприятий, к которым у него есть доступ. Администратор тоже может использовать меню Организатора.

Из карточки мероприятия Организатор может:

- посмотреть участников;
- найти запись по коду;
- отметить участника как пришедшего;
- закрыть регистрацию;
- закрыть мероприятие и отменить активные записи;
- изменить дату;
- изменить время;
- изменить место или ссылку;
- отправить ручное напоминание;
- пересобрать мероприятие через пошаговый редактор.

Создание и редактирование мероприятия работает как диалоговый конструктор. Черновик хранится в `organizer_states`, поэтому бот помнит текущий шаг между сообщениями.

## Участники и посещение

Список участников сортируется так, чтобы активные записи были выше отмененных. Для участника показываются имя, код записи и статус.

Организатор может отметить посещение:

- по кнопке в списке;
- через поиск по коду;
- через команду `/find КОД`.

Если запись уже отменена, отметить посещение нельзя. Если запись уже отмечена как посещенная, повторное действие не создает лишних изменений.

## Закрытие регистрации и мероприятия

Закрытие регистрации запрещает новые записи, но уже активные записи продолжают действовать.

Закрытие мероприятия жестче: Организатор закрывает мероприятие, активные записи получают статус отмены Организатором, а пользователям ставятся уведомления о закрытии.

Эти действия находятся в `OrganizerService.close_registration` и `OrganizerService.close_event`.

## Администрирование Организаторов

Администратор может назначить Организатора по MAX user id или ссылке вида `max://user/123`. Бот пытается извлечь числовой id.

Организатора можно снять через меню администратора, если он не задан конфигурацией. Пользователи из `ORGANIZER_USER_IDS` считаются управляемыми системой: чтобы убрать такую роль, нужно изменить конфигурацию и создать новую версию приложения.

Код: `AdminService`, административные ветки в `BotHandlers`.

## Ошибки для пользователя

Ожидаемые ошибки не показываются как traceback. Сервисный слой бросает доменные исключения из `app/domain.py`, а обработчик превращает их в короткие сообщения:

- “Свободных мест уже нет.”
- “Регистрация закрыта.”
- “Вы уже записаны на это мероприятие.”
- “У вас нет доступа к этому действию.”

Такой подход отделяет бизнес-правила от текста интерфейса и не смешивает обычные пользовательские ограничения с авариями приложения.
