# Производительность и метрики

Этот документ объясняет, как в проекте устроена диагностика задержек: что меряет приложение, как читать `perf_metric`, чем отличается внутренняя задержка бота от cold start Cloud Functions и куда смотреть при медленных кнопках.

Название файла историческое. Сейчас это не “аудит одной проблемы”, а рабочая документация по производительности.

## Главная модель задержки

Пользователь нажимает кнопку в MAX. Дальше происходит цепочка:

```text
MAX
  -> webhook Cloud Functions или FastAPI
  -> decode JSON
  -> dispatch_update
  -> BotHandlers
  -> Storage/YDB
  -> MAX API send_message
  -> HTTP 200 в MAX
```

Пользователь видит задержку как сумму нескольких вещей:

- доставка webhook от MAX до приложения;
- cold start serverless, если функция была холодной;
- разбор входящего события;
- бизнес-логика;
- чтения и записи YDB;
- исходящие вызовы MAX API;
- ожидание очереди отправки одному пользователю при быстрых повторных кликах.

Метрика `perf_metric` измеряет только работу внутри приложения. Она не включает весь путь MAX и не всегда включает cold start рантайма Cloud Functions.

## Где собираются метрики

Код:

```text
app/observability/performance.py
```

Метрики подключены в:

- `app/web.py` — FastAPI webhook;
- `app/function_handler.py` — Cloud Functions webhook и timer;
- `MeasuredStorage` — вызовы уровня хранилища;
- `MeasuredBotClient` — исходящие вызовы MAX API;
- `YdbStorage` — низкоуровневые YDB-вызовы.

Настройки:

```env
PERFORMANCE_METRICS_ENABLED=true
PERFORMANCE_METRICS_SLOW_MS=1000
```

Если `duration_ms` превышает `PERFORMANCE_METRICS_SLOW_MS`, поле `slow` будет `true`.

## Где смотреть логи

Yandex Cloud:

```powershell
yc serverless function logs max-bot --since 30m --limit 300
```

Ищите строки:

```json
{"event":"perf_metric", ...}
```

Локально при запуске через `uvicorn` метрики пишутся в stdout приложения.

## Поля `perf_metric`

Самые полезные поля:

| Поле | Что означает |
| --- | --- |
| `source` | Откуда вызов: `fastapi` или `cloud_function`. |
| `trigger` | Тип запуска: `webhook` или `timer`. |
| `update_type` | Тип события MAX: `bot_started`, `message_created`, `message_callback`. |
| `action` | Команда или кнопка: `/start`, `catalog`, `event_detail`, `my_regs` и так далее. |
| `duration_ms` | Полная длительность внутри приложения. |
| `slow` | Превышен ли порог `PERFORMANCE_METRICS_SLOW_MS`. |
| `decode_ms` | Разбор входящего JSON. |
| `dispatch_ms` | Основная бизнес-логика после decode. |
| `storage_calls` | Сколько методов хранилища вызвано. |
| `storage_ms` | Сколько времени занял слой хранилища. |
| `storage_methods` | Какие методы хранилища вызывались и сколько раз. |
| `ydb_calls` | Сколько низкоуровневых запросов к YDB выполнено. |
| `ydb_ms` | Сколько времени заняли YDB-запросы. |
| `ydb_methods` | Какие YDB-операции вызывались. |
| `max_calls` | Сколько исходящих вызовов MAX API выполнено. |
| `max_ms` | Сколько времени заняли вызовы MAX API. |
| `max_methods` | Какие методы MAX-клиента вызывались. |
| `send_lock_wait_ms` | Сколько ждали очередь отправки одному пользователю. |
| `input_media_count` | Сколько локальных картинок отправлено через fallback `InputMedia`. |
| `status_code` | HTTP-статус ответа webhook или timer. |
| `error_type` | Тип ошибки, если вызов упал. |

`storage_ms` и `ydb_ms` не нужно складывать. YDB-время входит внутрь времени storage-методов.

## Как читать метрику

Идите сверху вниз:

1. Смотрите `duration_ms`: был ли вызов реально медленным внутри приложения.
2. Смотрите `trigger`: webhook тормозит пользователя, timer тормозит фоновую отправку.
3. Смотрите `action`: какая кнопка или команда медленная.
4. Сравните `ydb_ms`, `max_ms` и `send_lock_wait_ms`.
5. Раскройте `storage_methods`, `ydb_methods` и `max_methods`.
6. Проверьте `input_media_count`.

Практические выводы:

- большой `ydb_ms` — узкое место в базе или слишком много запросов;
- большой `max_ms` — ожидание MAX API или загрузки вложений;
- большой `send_lock_wait_ms` — пользователь быстро нажал несколько кнопок, ответы встали в очередь;
- `input_media_count > 0` — бот отправляет локальные картинки, а не заранее загруженные token;
- `duration_ms` маленький, а пользователь жалуется на 3-4 секунды — вероятен cold start или внешняя доставка MAX.

## Cold start Cloud Functions

Cold start — холодный старт serverless-функции после простоя. Платформа заново поднимает runtime, импортирует Python-модули, создает объекты приложения, инициализирует YDB driver и MAX client.

В логах Cloud Functions cold start виден в строке `REPORT`:

```text
Function Init Duration: 3000 ms
```

Если рядом `perf_metric.duration_ms` равен `700`, а `REPORT Duration` выглядит сильно больше, значит пользователь ждал не только код бота, но и запуск функции.

Что влияет на cold start:

- размер зависимостей;
- импорт модулей Python;
- создание YDB driver;
- первичная очистка старых мероприятий;
- синхронизация ролей из настроек;
- память функции;
- частота вызовов.

Что можно сделать:

- не тащить лишние зависимости в пакет функции;
- не класть миграционные зависимости в runtime-пакет;
- держать инициализацию handler короткой;
- не запускать тяжелые операции при каждом webhook;
- принять, что serverless иногда будет холодным;
- если холодный старт критичен, рассмотреть свой сервер или другой режим прогрева.

## YDB-задержки

YDB Serverless хорошо подходит под событийный бот, но чувствителен к количеству последовательных round-trip.

Round-trip — это сетевой поход приложения в YDB и обратно. Десять маленьких запросов подряд могут быть заметнее, чем один более широкий запрос.

Типичные причины высокого `ydb_calls`:

- экран списка вызывает `get_event` по каждому мероприятию;
- для каждой записи отдельно читается пользователь, слот или мероприятие;
- универсальный метод грузит слоты и картинки там, где они не нужны;
- проверка роли делается несколькими запросами вместо одного `get_user_roles`;
- secondary index не используется через `VIEW`.

В проекте уже есть оптимизации:

- каталог читает мероприятия пачками;
- `my_regs` грузит связанные данные пачками;
- проверка активной записи использует `active_registration_keys`;
- главное меню получает роли одним вызовом;
- `touch_user` не читает пользователя перед записью;
- timer не делает полную repair-синхронизацию на каждом запуске.

Ориентиры не являются строгим SLA, но помогают заметить регресс:

| Действие | Нормальный смысл |
| --- | --- |
| `main_menu` | Несколько YDB-вызовов: пользователь, роли, состояние. |
| `catalog` | Не должен расти линейно по слотам каждого мероприятия. |
| `event_detail` | Может читать мероприятие, активную запись и диплинк. |
| `my_regs` | Должен грузить записи и связанные данные пачками. |
| обычный `timer` без due-уведомлений | Обычно мало YDB-вызовов. |
| repair `timer` | Может быть тяжелее, потому что синхронизирует напоминания. |

## MAX API-задержки

Высокий `max_ms` означает, что приложение ждет исходящий вызов MAX API. Причины:

- сеть или задержка MAX API;
- отправка вложений;
- повторная загрузка локальных картинок;
- слишком много сообщений в timer;
- лимитирование через `MAX_API_RPS`.

Для часто используемых картинок меню и уведомлений нужно использовать заранее загруженные MAX token. Документация: [max-image-assets.md](max-image-assets.md).

В норме:

```text
input_media_count = 0
```

Если значение больше нуля, бот использует fallback-отправку локальных PNG. Это допустимо локально, но плохо для продакшена.

## Очередь отправки одному пользователю

`send_lock_wait_ms` показывает, сколько обработчик ждал перед отправкой сообщения пользователю.

Зачем это нужно: если пользователь быстро нажимает несколько кнопок, ответы не должны перемешиваться. Поэтому отправка одному пользователю идет через lock внутри event loop.

Если `send_lock_wait_ms` большой, это не всегда ошибка приложения. Часто это быстрые повторные клики или долгое предыдущее отправление MAX API.

## Timer и уведомления

Timer trigger запускает `NotificationWorker`. Его метрики имеют `trigger=timer`.

Worker:

1. иногда запускает repair-синхронизацию напоминаний;
2. читает due-уведомления;
3. отправляет сообщения через MAX API;
4. меняет статус уведомления;
5. соблюдает `MAX_API_RPS`.

Настройки repair-синхронизации:

```env
REMINDER_SYNC_INTERVAL_MINUTES=60
REMINDER_SYNC_WINDOW_MINUTES=5
```

При этих значениях repair запускается в первые 5 минут каждого часа. Остальные timer-вызовы должны быть легче.

Отключить repair-синхронизацию:

```env
REMINDER_SYNC_INTERVAL_MINUTES=0
```

## Быстрая сводка по логам

PowerShell:

```powershell
$lines = yc serverless function logs max-bot --since 30m --limit 500
$metrics = foreach ($line in $lines) {
    if ($line -match '\{.*"event":\s*"perf_metric".*\}$') {
        $Matches[0] | ConvertFrom-Json
    }
}

$metrics |
    Group-Object trigger,action |
    ForEach-Object {
        $items = $_.Group
        [pscustomobject]@{
            action = $_.Name
            count = $items.Count
            avg_ms = [math]::Round(($items.duration_ms | Measure-Object -Average).Average, 1)
            max_ms = [math]::Round(($items.duration_ms | Measure-Object -Maximum).Maximum, 1)
            ydb_avg = [math]::Round(($items.ydb_ms | Measure-Object -Average).Average, 1)
            max_avg = [math]::Round(($items.max_ms | Measure-Object -Average).Average, 1)
            ydb_calls_avg = [math]::Round(($items.ydb_calls | Measure-Object -Average).Average, 1)
            slow = @($items | Where-Object { $_.slow -eq $true }).Count
        }
    } |
    Sort-Object action |
    Format-Table -AutoSize
```

Это дает грубую картину: какие действия часто вызываются, какие медленные, где среднее YDB-время, где MAX-время.

## Исторический контекст

Изначально заметная задержка кнопок была связана с отправкой локальных PNG через `InputMedia`: библиотека `maxapi` после загрузки медиа делала паузу перед отправкой сообщения. Поэтому в проекте появилась отдельная схема с заранее загруженными MAX token.

Эта история важна как урок, но не как смысл документации. Текущее правило простое: часто используемые картинки должны идти по `token`, а `input_media_count` в продакшене должен быть `0`.

## Что делать при медленной кнопке

1. Найдите `perf_metric` по `action`.
2. Сравните `duration_ms` с пользовательской задержкой.
3. Если в `REPORT` есть большой `Function Init Duration`, учитывайте cold start.
4. Если большой `ydb_ms`, смотрите `ydb_calls` и `ydb_methods`.
5. Если большой `max_ms`, смотрите `max_methods` и `input_media_count`.
6. Если большой `send_lock_wait_ms`, проверьте повторные клики и предыдущую отправку.
7. После изменения прогоните тесты и проверьте метрики на реальном сценарии.

## Когда пора менять архитектуру

Yandex serverless подходит для текущего сценария, но не для всего на свете.

Стоит подумать о своем сервере или другой архитектуре, если:

- cold start недопустим даже редко;
- нужен долгоживущий фоновый процесс;
- timer должен работать чаще и предсказуемее;
- нагрузка стала постоянной и serverless перестал быть дешевым;
- нужна более сложная наблюдаемость, чем логи Cloud Functions.

До этого момента дешевле и проще оптимизировать конкретные действия: уменьшать YDB round-trip, отправлять картинки по token и держать webhook коротким.
