# Аудит производительности webhook и кнопок

Дата аудита: 2026-05-25.

## Главный вывод

Основная видимая задержка после нажатия кнопок возникает не из-за одной большой операции, а из-за синхронной цепочки внутри webhook: обработчик ждёт чтения и записи в YDB, загрузки картинки в MAX и вызова MAX API, и только потом возвращает HTTP-ответ.

Самая явная причина задержки в 2-3 секунды найдена в работе с локальными картинками меню. Проект передаёт PNG-файлы через `local_image_attachment(...)`, а библиотека `maxapi` после загрузки такого медиа специально делает паузу `after_input_media_delay`, по умолчанию `2.0` секунды, перед отправкой или редактированием сообщения.

## Где именно возникает задержка

Цепочка обработки входящего webhook:

1. `app/function_handler.py` или `app/web.py` принимает событие.
2. `dispatch_update(...)` в `app/bot/dispatcher.py` вызывает `BotHandlers`.
3. `BotHandlers.handle_callback(...)` в `app/bot/handlers.py` выполняет бизнес-логику.
4. `_send(...)` в `app/bot/handlers.py` ждёт `edit_message(...)` или `send_message(...)`.
5. `MaxApiBotClient` в `app/bot/client.py` вызывает библиотеку `maxapi`.
6. Если вложение является `InputMedia`, `maxapi` загружает файл, затем делает `await asyncio.sleep(bot.after_input_media_delay)`.

Значение по умолчанию задано в `maxapi/maxapi/bot.py`:

```python
self.after_input_media_delay = after_input_media_delay or 2.0
```

Пауза используется в:

- `maxapi/maxapi/methods/send_message.py`;
- `maxapi/maxapi/methods/edit_message.py`.

## Кнопки и действия, которые попадают в задержку 2 секунды

Задержка проявляется там, где ответ содержит локальную картинку через `local_image_attachment(...)`:

- главное меню: `MAIN_MENU_IMAGE_PATH`, картинка `app/assets/main-menu.png`;
- каталог мероприятий: та же `main-menu.png`;
- меню организатора: `app/assets/organizer-menu.png`;
- список участников: `app/assets/participants-menu.png`;
- уведомления из timer worker: `app/assets/notification-reminder.png`, это не кнопка, но тот же механизм.

Размеры локальных картинок на момент аудита:

- `main-menu.png` — около 1.2 МБ;
- `organizer-menu.png` — около 1.7 МБ;
- `participants-menu.png` — около 390 КБ;
- `notification-reminder.png` — около 111 КБ.

Из-за этого нажатия вроде «Главное меню», «Мероприятия», «Меню организатора», «Участники», перелистывание каталога и возврат в эти экраны почти гарантированно получают минимум около 2 секунд дополнительного ожидания. Если сверху добавляется сетевой вызов MAX и запросы к YDB, пользователь и видит 2-3 секунды.

## Второй источник задержек: много мелких запросов в YDB

Webhook сейчас обрабатывается синхронно: HTTP-ответ MAX возвращается только после завершения всей логики. Внутри одного нажатия выполняются обращения к хранилищу, а в `YdbStorage` многие методы раскрываются в несколько запросов.

Примеры:

- `upsert_user(...)` сначала читает пользователя, потом делает `UPSERT`: минимум 2 операции YDB на каждое сообщение и callback.
- `list_events(...)` читает мероприятия, затем для каждого мероприятия отдельно читает слоты и картинку: `1 + 2 * N` запросов.
- `available_places(...)` заново читает мероприятие, его слоты и картинку; для мероприятия со слотами рекурсивно делает это по слотам.
- `get_event(...)` читает мероприятие, слоты и картинку: около 3 запросов.
- `list_user_registrations(...)` для каждой записи дополнительно подтягивает мероприятие, слот и пользователя.
- `create_registration(...)` делает транзакцию, генерирует id через отдельные проверки, пишет запись, код, активный ключ, счётчики, напоминания и аудит, затем заново читает созданную запись.

На тестовом подсчёте вызовов уровня `Storage` для одного callback получились такие числа:

| Действие | Вызовы `Storage` |
| --- | ---: |
| `catalog` | 11 |
| `event_detail` | 6 |
| `event_book` со слотами | 5 |
| `register_confirm` | 3, но внутри `create_registration` много YDB-операций |
| `my_regs` | 3 |
| `org_menu` | 4 |
| `org_event` | 7 |
| `org_participants` | 6 |

Это не точное число YDB-запросов, потому что один вызов `Storage` может раскрыться в несколько сетевых операций. Например, каталог с несколькими мероприятиями может легко превратиться в десятки запросов.

## Холодный старт

После простоя Cloud Function отдельно возможен холодный старт. В проекте при создании handler выполняется:

- создание `YdbStorage` и `driver.wait(timeout=10)`;
- первичная очистка старых мероприятий;
- синхронизация ролей из настроек.

Это влияет на первый запрос после простоя. Но повторяемая задержка после конкретных кнопок лучше объясняется не холодным стартом, а загрузкой локальных картинок плюс синхронными запросами.

## Что делать в первую очередь

1. Убрать загрузку локальных PNG из интерактивных меню или заменить её на заранее загруженные `token` MAX. Это самый быстрый путь убрать стабильные 2 секунды.
2. Если картинки нужны, не передавать `InputMedia` на каждый ответ. Предзагрузить медиа один раз и хранить token в конфигурации или базе.
3. В `MaxApiBotClient` явно настроить `after_input_media_delay`, если после проверки MAX API допускает меньшую паузу. Но это менее надёжно, чем не загружать картинку каждый раз.
4. Оптимизировать каталог: не вызывать `available_places(...)` отдельно для каждого мероприятия после `list_events(...)`, а отдавать нужные поля одним запросом или использовать уже загруженные `booked_count` и слоты.
5. Сократить повторные чтения `event_images` и `event_slots`: сейчас они подтягиваются почти при каждом `get_event(...)`.
6. Добавить измерение времени по шагам: webhook decode, storage, MAX upload, MAX edit/send. Без этого в логах задержки выглядят как «бот думает», хотя на деле он ждёт конкретные внешние операции.

## Метрики после оптимизации

Бот пишет одну JSON-строку на каждый webhook и timer-вызов Cloud Functions:

```powershell
yc serverless function logs max-bot --since 30m --limit 200
```

Ищите строки с `"event":"perf_metric"`. Самые полезные поля:

- `action` — кнопка или команда, например `catalog`, `event_detail`, `/events`;
- `duration_ms` и `slow` — полное время обработки и признак превышения порога;
- `dispatch_ms` — время бизнес-логики после разбора webhook;
- `storage_calls`, `storage_ms`, `storage_methods` — вызовы уровня хранилища;
- `ydb_calls`, `ydb_ms`, `ydb_methods` — реальные низкоуровневые запросы YDB;
- `max_calls`, `max_ms`, `max_methods` — отправка, редактирование и удаление сообщений в MAX;
- `send_lock_wait_ms` — ожидание очереди отправки для одного пользователя;
- `input_media_count` — число fallback-картинок через `InputMedia`. В норме должно быть `0`, потому что основные картинки должны идти по MAX token.

Порог `slow` настраивается переменной `PERFORMANCE_METRICS_SLOW_MS`, по умолчанию `1000`. Метрики можно выключить через `PERFORMANCE_METRICS_ENABLED=false`, но для диагностики медленных кнопок лучше держать их включёнными: это один структурированный лог на входящий вызов без записи в YDB.

## Как читать `perf_metric`

`perf_metric` — это внутренняя длительность обработки внутри кода бота. Она не равна строке Cloud Functions `REPORT Duration`, потому что `REPORT` включает холодный старт рантайма. Если в `REPORT` видно `Function Init Duration: 3000 ms`, а в `perf_metric.duration_ms` около `800`, значит пользовательский код работал около `800` мс, а остальное съел запуск Serverless.

Быстрый просмотр последних строк:

```powershell
yc serverless function logs max-bot --since 30m --limit 300
```

Сводка по действиям:

```powershell
$lines = yc serverless function logs max-bot --since 30m --limit 500
$metrics = foreach ($line in $lines) {
    if ($line -match '\{.*"event":\s*"perf_metric".*\}$') {
        $Matches[0] | ConvertFrom-Json
    }
}
$metrics |
    Group-Object trigger,action |
    ForEach-Object {
        $items = $_.Group
        [pscustomobject]@{
            action = $_.Name
            count = $items.Count
            avg_ms = [math]::Round(($items.duration_ms | Measure-Object -Average).Average, 1)
            max_ms = [math]::Round(($items.duration_ms | Measure-Object -Maximum).Maximum, 1)
            ydb_avg = [math]::Round(($items.ydb_ms | Measure-Object -Average).Average, 1)
            max_avg = [math]::Round(($items.max_ms | Measure-Object -Average).Average, 1)
            ydb_calls_avg = [math]::Round(($items.ydb_calls | Measure-Object -Average).Average, 1)
            slow = @($items | Where-Object { $_.slow -eq $true }).Count
        }
    } |
    Sort-Object action |
    Format-Table -AutoSize
```

Читать строку лучше сверху вниз:

- `trigger` показывает источник: `webhook` — пользователь нажал кнопку или написал команду, `timer` — фоновая отправка уведомлений.
- `action` показывает кнопку или команду. Для нажатий на встроенные кнопки это имя из `Payload`, например `main_menu`, `catalog`, `event_detail`, `my_regs`.
- `duration_ms` — главное число: сколько заняла обработка внутри бота.
- `slow=true` означает, что `duration_ms` превысил `PERFORMANCE_METRICS_SLOW_MS`.
- `dispatch_ms` — почти вся бизнес-логика после разбора webhook. Для кнопок обычно близко к `duration_ms`.
- `storage_ms` и `storage_methods` показывают, какие методы хранилища были дорогими. Это уровень приложения: например `list_user_registrations`.
- `ydb_ms`, `ydb_calls`, `ydb_methods` показывают реальные запросы к YDB внутри этих методов. `storage_ms` и `ydb_ms` не надо складывать: YDB уже входит внутрь `storage`.
- `max_ms`, `max_calls`, `max_methods` показывают время API MAX. Если тут сотни миллисекунд или секунды, база уже не главный виновник.
- `input_media_count` должен быть `0`. Если больше нуля, бот снова отправляет локальные картинки через fallback-загрузку.
- `send_lock_wait_ms` показывает очередь отправки одному пользователю. Большое значение обычно означает быстрые повторные клики, которые сами выстроили очередь.

Практическое правило простое: сначала смотрим `duration_ms`, потом ищем самый большой вклад между `ydb_ms`, `max_ms` и `send_lock_wait_ms`. После этого раскрываем `storage_methods`, `ydb_methods` или `max_methods`, чтобы увидеть конкретный метод.

Свежий срез после оптимизации YDB-чтений 2026-05-25:

| Действие | `duration_ms` | `ydb_calls` | `ydb_ms` | `max_ms` | Вывод |
| --- | ---: | ---: | ---: | ---: | --- |
| `main_menu` | 755 | 7 | 655 | 97 | Основная тёплая цена — YDB: очистка временных состояний, проверка роли и запись `last_bot_message_id`. |
| `my_regs` | 852 | 9 | 744 | 104 | В пределах ориентира `8-12` YDB-вызовов; узкое место — чтение списка записей и пакетная сборка связанных данных. |
| `timer` | 375-1003 | 8 | 374-1002 | 0 | N+1 по регистрациям убран: цикл держится на 8 YDB-вызовах, но всё ещё почти полностью ждёт YDB. |

По этим цифрам MAX сейчас не является главным тормозом для проверенных действий: отправка/редактирование занимает около `100` мс. Оставшаяся тёплая задержка почти целиком в YDB, плюс отдельно Cloud Functions часто добавляет холодный старт около `3-3.6` секунды в `REPORT`.

## Снижение обязательных YDB-вызовов

Следующая оптимизация убирает не N+1, а базовые round-trip'ы, которые раньше выполнялись почти на каждый webhook:

- `touch_user(...)` заменяет полный `upsert_user(...)` в webhook и не читает пользователя перед записью;
- главное меню читает все роли пользователя одним `get_user_roles(...)`, вместо отдельных проверок Организатора и админа;
- после успешного редактирования того же сообщения бот не пишет `last_bot_message_id` повторно;
- переходы в главное меню и команды `/organizer`, `/admin` чистят черновики одним `clear_user_draft_state(...)`;
- `my_regs` грузит только данные для списка записей: записи и мероприятия без слотов, картинок и пользователя;
- обычный timer сначала ищет due-уведомления, а полную repair-синхронизацию напоминаний запускает только в sync-окне.

Настройки repair-синхронизации:

- `REMINDER_SYNC_INTERVAL_MINUTES`, по умолчанию `60`;
- `REMINDER_SYNC_WINDOW_MINUTES`, по умолчанию `5`;
- `REMINDER_SYNC_INTERVAL_MINUTES=0` отключает repair-синхронизацию в timer.

При стандартных значениях timer, который срабатывает в первые 5 минут часа, будет иметь поведение `repair timer` и может оставаться около `8 ydb_calls`. Остальные срабатывания являются обычным timer: если due-уведомлений нет, ориентир `1-2 ydb_calls`.

Новые ориентиры после деплоя:

- `main_menu`: около `3-4` `ydb_calls`;
- `my_regs`: около `4-5` `ydb_calls`;
- обычный `timer`: около `1-2` `ydb_calls`;
- `repair timer`: около текущих `8` `ydb_calls`.

## Оптимизация YDB-чтений после метрик

После удаления обязательной паузы на загрузку локальных картинок оставшаяся тёплая задержка в основном приходила из YDB. Метрики показали типичную картину не одной тяжёлой операции, а множества коротких сетевых запросов подряд: каталог тянул слоты по каждому мероприятию, карточка мероприятия искала активную запись через полный список записей пользователя, `my_regs` подтягивал мероприятие, слот и пользователя отдельно для каждой записи, а timer-обработчик синхронизации напоминаний ходил за регистрациями и outbox-строками в цикле.

Причина такого кода понятна: универсальные методы вроде `get_event(...)`, `get_registration(...)` и `list_user_registrations(...)` удобны и безопасны для раннего MVP, потому что каждый экран получает уже «богатый» объект со всеми связанными данными. Минус проявился только после роста сценариев: один красивый вызов уровня `Storage` раскрывался в пачку запросов к YDB. Важной бизнес-логики в этих лишних чтениях не было, это скорее осторожная универсальность, которая стала дорогой.

Что изменено без изменения схемы YDB:

- `list_events(...)` читает мероприятия одним запросом, а слоты и картинки подтягивает пачками.
- `event_detail` проверяет активную запись через `get_active_registration_for_event(...)` и таблицу `active_registration_keys`, без полного списка записей пользователя.
- `list_user_registrations(...)` сначала читает записи пользователя, затем пачкой подтягивает мероприятия, слоты и пользователя.
- `list_organizer_events(...)` для организатора берёт связи `organizer_events`, затем пачкой читает мероприятия.
- Timer-синхронизация напоминаний читает активные регистрации, мероприятия, слоты и существующие outbox-строки пачками.
- Запросы по существующим вторичным индексам явно используют `VIEW`, как рекомендует документация YDB для выборок по secondary index: https://ydb.tech/docs/ru/yql/reference/syntax/select/secondary_index.

Ориентиры после деплоя на малом наборе данных:

- `catalog`: примерно до `6-8` `ydb_calls`, без роста на каждое мероприятие из-за слотов.
- `event_detail`: примерно до `8-10` `ydb_calls`, без `list_user_registrations(...)`.
- `my_regs`: примерно до `8-12` `ydb_calls`, без чтения мероприятия/слота/пользователя по каждой записи.
- `timer`: пустой или почти пустой цикл не должен давать десятки фиксированных запросов; для большого числа записей рост должен быть пачками, а не по одной регистрации.

Проверка после деплоя:

```powershell
yc serverless function logs max-bot --since 30m --limit 300
```

В строках `"event":"perf_metric"` смотрите `action`, `duration_ms`, `ydb_calls`, `ydb_ms`, `ydb_methods`, `storage_methods` и `max_ms`. Если `slow=true`, сначала разделите задержку на YDB и MAX: высокий `ydb_ms` означает проблему чтений/записей в базе, высокий `max_ms` — ожидание API MAX, высокий `send_lock_wait_ms` — очередь быстрых кликов одного пользователя.
