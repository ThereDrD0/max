# Аудит производительности webhook и кнопок

Дата аудита: 2026-05-25.

## Главный вывод

Основная видимая задержка после нажатия кнопок возникает не из-за одной большой операции, а из-за синхронной цепочки внутри webhook: обработчик ждёт чтения и записи в YDB, загрузки картинки в MAX и вызова MAX API, и только потом возвращает HTTP-ответ.

Самая явная причина задержки в 2-3 секунды найдена в работе с локальными картинками меню. Проект передаёт PNG-файлы через `local_image_attachment(...)`, а библиотека `maxapi` после загрузки такого медиа специально делает паузу `after_input_media_delay`, по умолчанию `2.0` секунды, перед отправкой или редактированием сообщения.

## Где именно возникает задержка

Цепочка обработки входящего webhook:

1. `app/function_handler.py` или `app/web.py` принимает событие.
2. `dispatch_update(...)` в `app/bot/dispatcher.py` вызывает `BotHandlers`.
3. `BotHandlers.handle_callback(...)` в `app/bot/handlers.py` выполняет бизнес-логику.
4. `_send(...)` в `app/bot/handlers.py` ждёт `edit_message(...)` или `send_message(...)`.
5. `MaxApiBotClient` в `app/bot/client.py` вызывает библиотеку `maxapi`.
6. Если вложение является `InputMedia`, `maxapi` загружает файл, затем делает `await asyncio.sleep(bot.after_input_media_delay)`.

Значение по умолчанию задано в `maxapi/maxapi/bot.py`:

```python
self.after_input_media_delay = after_input_media_delay or 2.0
```

Пауза используется в:

- `maxapi/maxapi/methods/send_message.py`;
- `maxapi/maxapi/methods/edit_message.py`.

## Кнопки и действия, которые попадают в задержку 2 секунды

Задержка проявляется там, где ответ содержит локальную картинку через `local_image_attachment(...)`:

- главное меню: `MAIN_MENU_IMAGE_PATH`, картинка `app/assets/main-menu.png`;
- каталог мероприятий: та же `main-menu.png`;
- меню организатора: `app/assets/organizer-menu.png`;
- список участников: `app/assets/participants-menu.png`;
- уведомления из timer worker: `app/assets/notification-reminder.png`, это не кнопка, но тот же механизм.

Размеры локальных картинок на момент аудита:

- `main-menu.png` — около 1.2 МБ;
- `organizer-menu.png` — около 1.7 МБ;
- `participants-menu.png` — около 390 КБ;
- `notification-reminder.png` — около 111 КБ.

Из-за этого нажатия вроде «Главное меню», «Мероприятия», «Меню организатора», «Участники», перелистывание каталога и возврат в эти экраны почти гарантированно получают минимум около 2 секунд дополнительного ожидания. Если сверху добавляется сетевой вызов MAX и запросы к YDB, пользователь и видит 2-3 секунды.

## Второй источник задержек: много мелких запросов в YDB

Webhook сейчас обрабатывается синхронно: HTTP-ответ MAX возвращается только после завершения всей логики. Внутри одного нажатия выполняются обращения к хранилищу, а в `YdbStorage` многие методы раскрываются в несколько запросов.

Примеры:

- `upsert_user(...)` сначала читает пользователя, потом делает `UPSERT`: минимум 2 операции YDB на каждое сообщение и callback.
- `list_events(...)` читает мероприятия, затем для каждого мероприятия отдельно читает слоты и картинку: `1 + 2 * N` запросов.
- `available_places(...)` заново читает мероприятие, его слоты и картинку; для мероприятия со слотами рекурсивно делает это по слотам.
- `get_event(...)` читает мероприятие, слоты и картинку: около 3 запросов.
- `list_user_registrations(...)` для каждой записи дополнительно подтягивает мероприятие, слот и пользователя.
- `create_registration(...)` делает транзакцию, генерирует id через отдельные проверки, пишет запись, код, активный ключ, счётчики, напоминания и аудит, затем заново читает созданную запись.

На тестовом подсчёте вызовов уровня `Storage` для одного callback получились такие числа:

| Действие | Вызовы `Storage` |
| --- | ---: |
| `catalog` | 11 |
| `event_detail` | 6 |
| `event_book` со слотами | 5 |
| `register_confirm` | 3, но внутри `create_registration` много YDB-операций |
| `my_regs` | 3 |
| `org_menu` | 4 |
| `org_event` | 7 |
| `org_participants` | 6 |

Это не точное число YDB-запросов, потому что один вызов `Storage` может раскрыться в несколько сетевых операций. Например, каталог с несколькими мероприятиями может легко превратиться в десятки запросов.

## Холодный старт

После простоя Cloud Function отдельно возможен холодный старт. В проекте при создании handler выполняется:

- создание `YdbStorage` и `driver.wait(timeout=10)`;
- первичная очистка старых мероприятий;
- синхронизация ролей из настроек.

Это влияет на первый запрос после простоя. Но повторяемая задержка после конкретных кнопок лучше объясняется не холодным стартом, а загрузкой локальных картинок плюс синхронными запросами.

## Что делать в первую очередь

1. Убрать загрузку локальных PNG из интерактивных меню или заменить её на заранее загруженные `token` MAX. Это самый быстрый путь убрать стабильные 2 секунды.
2. Если картинки нужны, не передавать `InputMedia` на каждый ответ. Предзагрузить медиа один раз и хранить token в конфигурации или базе.
3. В `MaxApiBotClient` явно настроить `after_input_media_delay`, если после проверки MAX API допускает меньшую паузу. Но это менее надёжно, чем не загружать картинку каждый раз.
4. Оптимизировать каталог: не вызывать `available_places(...)` отдельно для каждого мероприятия после `list_events(...)`, а отдавать нужные поля одним запросом или использовать уже загруженные `booked_count` и слоты.
5. Сократить повторные чтения `event_images` и `event_slots`: сейчас они подтягиваются почти при каждом `get_event(...)`.
6. Добавить измерение времени по шагам: webhook decode, storage, MAX upload, MAX edit/send. Без этого в логах задержки выглядят как «бот думает», хотя на деле он ждёт конкретные внешние операции.

## Метрики после оптимизации

Бот пишет одну JSON-строку на каждый webhook и timer-вызов Cloud Functions:

```powershell
yc serverless function logs max-bot --since 30m --limit 200
```

Ищите строки с `"event":"perf_metric"`. Самые полезные поля:

- `action` — кнопка или команда, например `catalog`, `event_detail`, `/events`;
- `duration_ms` и `slow` — полное время обработки и признак превышения порога;
- `dispatch_ms` — время бизнес-логики после разбора webhook;
- `storage_calls`, `storage_ms`, `storage_methods` — вызовы уровня хранилища;
- `ydb_calls`, `ydb_ms`, `ydb_methods` — реальные низкоуровневые запросы YDB;
- `max_calls`, `max_ms`, `max_methods` — отправка, редактирование и удаление сообщений в MAX;
- `send_lock_wait_ms` — ожидание очереди отправки для одного пользователя;
- `input_media_count` — число fallback-картинок через `InputMedia`. В норме должно быть `0`, потому что основные картинки должны идти по MAX token.

Порог `slow` настраивается переменной `PERFORMANCE_METRICS_SLOW_MS`, по умолчанию `1000`. Метрики можно выключить через `PERFORMANCE_METRICS_ENABLED=false`, но для диагностики медленных кнопок лучше держать их включёнными: это один структурированный лог на входящий вызов без записи в YDB.
