# Архитектура проекта

Этот документ объясняет, как устроен проект в коде. Он рассчитан на человека, который впервые открыл репозиторий и пока не знает, где искать бизнес-логику, где webhook, где база и почему часть работы вынесена в timer.

## Общая идея

Проект — это Python-приложение для MAX-бота. Снаружи оно принимает webhook-события MAX, внутри превращает их в пользовательские действия, вызывает сервисный слой и сохраняет состояние в хранилище. В продакшене хранилище — YDB Serverless.

Главный поток выглядит так:

```text
MAX
  -> HTTP webhook
  -> app.function_handler или app.web
  -> app.bot.dispatcher.dispatch_update
  -> app.bot.handlers.BotHandlers
  -> app.services.*
  -> app.storage.*
  -> YDB
  -> MAX API для ответа пользователю
```

Есть два режима запуска:

- локальный или серверный FastAPI: `app/main.py` создает `app.web.create_app`;
- Yandex Cloud Functions: корневой `index.py` экспортирует `handler = create_function_handler()`.

Оба режима используют один и тот же dispatcher, сервисы, хранилище и MAX-клиент. Отличается только внешний HTTP-слой.

## Входные точки

`app/main.py` нужен для `uvicorn app.main:app`. Он почти пустой: импортирует `app_factory()` из `app/web.py`.

`app/web.py` создает FastAPI-приложение:

- `GET /healthz` отвечает, что процесс жив;
- `GET /readyz` дополнительно проверяет доступность хранилища;
- `POST /webhook` принимает событие MAX, проверяет `WEBHOOK_SECRET`, разбирает JSON и передает событие в `dispatch_update`.

`index.py` нужен Yandex Cloud Functions. Он отдает наружу `handler`, созданный в `app/function_handler.py`.

`app/function_handler.py` делает то же, что FastAPI-слой, но в формате Cloud Functions:

- `GET` возвращает health JSON;
- `POST` принимает webhook MAX;
- timer event запускает отправку уведомлений;
- `_AsyncRunner` держит постоянный асинхронный event loop, чтобы клиент MAX не ломался между вызовами функции.

Event loop — это цикл событий Python для асинхронного кода. Если каждый webhook запускать через `asyncio.run`, цикл закрывается после запроса, а HTTP-сессия библиотеки MAX может остаться привязанной к закрытому циклу. Поэтому в Cloud Functions используется постоянный `_AsyncRunner`.

## Разбор событий MAX

`app/bot/dispatcher.py` получает сырой словарь update и выбирает обработчик по `update_type`:

- `bot_started` — пользователь запустил бота или перешел по диплинку;
- `message_created` — пользователь отправил текст;
- `message_callback` — пользователь нажал inline-кнопку.

Dispatcher не содержит бизнес-логики. Его задача — достать `user_id`, имя, `chat_id`, текст или payload кнопки и вызвать метод `BotHandlers`.

Если MAX пришлет неизвестный тип события, проект его игнорирует. Это нормальное поведение для webhook: бот отвечает успешно на ненужные события, чтобы MAX не повторял доставку.

## Обработчики бота

Основной файл пользовательского поведения — `app/bot/handlers.py`.

В нем находится `BotHandlers`, который:

- показывает дисклеймер и главное меню;
- строит каталог мероприятий;
- показывает карточку мероприятия;
- ведет запись и отмену записи;
- показывает мои записи;
- открывает меню Организатора;
- ведет создание и редактирование мероприятия;
- показывает участников;
- запускает поиск записи по коду;
- открывает меню администратора;
- превращает ожидаемые доменные ошибки в понятные сообщения пользователю.

Payload кнопок упаковывается и распаковывается в `app/bot/payloads.py`. Payload — это короткая строка, которую MAX возвращает боту при нажатии кнопки. В проекте она выглядит как:

```text
action|event_id|slot_id|registration_id|value
```

Кнопки собираются в `app/bot/keyboards.py`. Диплинки на мероприятия — в `app/bot/deeplinks.py`.

## Сервисный слой

Сервисный слой лежит в `app/services`. Он отделяет бизнес-правила от деталей интерфейса MAX и от конкретной базы данных.

`RegistrationService` отвечает за обычного пользователя:

- согласие на обработку минимальных данных;
- список будущих мероприятий;
- расчет свободных мест;
- создание записи;
- отмена записи;
- включение и отключение уведомлений;
- список записей пользователя.

`OrganizerService` отвечает за Организатора:

- список доступных мероприятий;
- создание и обновление мероприятия;
- перенос даты и смену места;
- закрытие регистрации и закрытие мероприятия;
- поиск записи по коду;
- отметку посещения;
- постановку ручных уведомлений в очередь.

`AdminService` отвечает за роль администратора:

- проверка доступа к админ-меню;
- назначение роли Организатора;
- снятие роли Организатора;
- защита Организаторов, заданных через `ORGANIZER_USER_IDS`.

`NotificationWorker` отправляет уведомления из `notification_outbox`. Он запускается не из пользовательского webhook, а timer-событием. Это важно: массовая отправка не должна держать ответ MAX на пользовательское нажатие.

`EventCleanupService` удаляет старые мероприятия и отмененные пользователем записи по правилам хранения. В FastAPI он запускается при старте приложения, в Cloud Functions — при инициализации и затем периодически через scheduler внутри handler.

## Доменные ошибки

`app/domain.py` содержит ожидаемые бизнес-ошибки: нет согласия, нет мест, регистрация закрыта, нет доступа, запись не найдена и так далее.

Это не аварии приложения. Это нормальные ситуации, которые надо показать пользователю понятным текстом. `BotHandlers._friendly_error` переводит такие исключения в сообщения интерфейса.

## Модель данных в коде

Основные сущности описаны dataclass-классами в `app/storage/entities.py`:

- `User` — пользователь MAX;
- `Consent` — согласие на обработку минимальных данных;
- `Event` — мероприятие;
- `EventSlot` — слот мероприятия;
- `Registration` — запись пользователя;
- `RoleAssignment` — роль пользователя;
- `OrganizerState` — черновик диалога Организатора;
- `NotificationOutbox` — строка очереди уведомлений;
- `AuditLog` — запись аудита.

Эти классы не являются ORM-моделями. ORM — это слой, который автоматически связывает объекты Python с таблицами базы. В проекте ORM нет: хранилище явно читает и пишет YDB через SDK.

## Хранилище

Контракт хранилища описан в `app/storage/base.py` как `Protocol`. Это интерфейс: какие методы нужны сервисам, без привязки к YDB.

Реализации:

- `app/storage/ydb.py` — основное хранилище для локальной YDB и продакшена;
- `app/storage/memory.py` — память процесса для тестов;
- `app/storage/factory.py` — выбирает реализацию по `STORAGE_BACKEND`.

Сервисный слой работает через интерфейс `Storage`, поэтому тесты могут использовать память, а продакшен — YDB.

## YDB-схема

Схема YDB задается в `app/ydb_schema.py`. Команда:

```bash
python -m app.ydb_schema
```

создает таблицы и индексы идемпотентно: повторный запуск не должен ломать существующую базу. Это не полноценная система миграций с версиями, а текущий способ поддерживать схему для serverless-развертывания.

Подробности про таблицы, индексы, транзакции и отличия от PostgreSQL описаны в [docs/ydb.md](ydb.md).

## Seed-данные

Стартовые мероприятия лежат в `seed/events.yaml`. Загрузка:

```bash
python -m app.seed
```

Seed создает мероприятия идемпотентно по паре `(title, starts_at)`, чтобы повторный запуск не плодил дубликаты. Организаторов можно назначать через `ORGANIZER_USER_IDS` или через поле `organizer_user_ids` конкретного мероприятия в seed-файле.

## Уведомления

Уведомления не отправляются сразу длинным циклом из пользовательского webhook. Вместо этого проект пишет задачи в `notification_outbox`.

`NotificationWorker` выбирает уведомления со статусом `pending`, у которых `send_after <= now`, отправляет сообщение через MAX API и меняет статус на `sent`, `failed` или `skipped`.

Автоматические напоминания создаются при записи на мероприятие и синхронизируются repair-логикой. Ручные уведомления ставит Организатор через меню.

## Наблюдаемость

`app/observability/performance.py` собирает метрики одного вызова webhook или timer. Метрика печатается одной JSON-строкой с событием `perf_metric`.

Важные поля:

- `duration_ms` — сколько занял вызов внутри приложения;
- `storage_calls` и `storage_ms` — вызовы уровня хранилища;
- `ydb_calls` и `ydb_ms` — реальные обращения к YDB;
- `max_calls` и `max_ms` — обращения к MAX API;
- `send_lock_wait_ms` — ожидание очереди отправки одному пользователю;
- `input_media_count` — отправка локальных картинок через запасной путь.

Подробная расшифровка — в [docs/performance-audit.md](performance-audit.md).

## Почему архитектура такая

Webhook должен быстро вернуть MAX успешный HTTP-ответ. Поэтому проект держит пользовательские действия короткими, а массовую работу выносит в очередь и timer.

YDB выбрана как serverless-хранилище под Cloud Functions: не надо держать отдельную виртуальную машину с базой, но приходится явно думать о ключах, индексах, количестве сетевых round-trip и транзакциях.

Слой `Storage` нужен, чтобы бизнес-правила не знали, как именно устроена база. Это делает тесты проще и снижает риск, что интерфейс MAX, сервисы и YDB-детали смешаются в один большой файл.
