# Работа с git

Этот проект использует локальный git-репозиторий. Основная и единственная рабочая ветка - `master`.

Перед любой git-командой ИИ должен перечитать этот файл и действовать по правилам ниже. Эти правила относятся только к работе с git и не меняют порядок обычной разработки.

## Ветки

Работа ведется только в ветке `master`, потому что сейчас над проектом работает один человек.

Не создавай рабочие ветки без отдельной просьбы пользователя. Если команда или инструмент предлагает перейти на другую ветку, сначала остановись и уточни необходимость.

## Запрещенные действия

Нельзя использовать команды, которые изменяют историю коммитов или удаляют коммиты. В частности, запрещены:

- `git reset --hard`;
- `git rebase`;
- `git commit --amend`;
- `git filter-branch`;
- `git filter-repo`;
- `git push --force`;
- `git reflog expire`;
- удаление веток или тегов без прямой просьбы пользователя.

Если нужна операция, похожая на переписывание истории, сначала объясни риск и попроси отдельное подтверждение. По умолчанию выбирай безопасный путь через новый коммит.

## Коммиты

Каждый коммит должен быть сделан в формате Conventional Commits - это соглашение, где первая строка коротко описывает тип изменения:

```text
type(scope): краткое описание
```

Для этого проекта чаще всего подходят такие типы:

- `feat` - новая возможность;
- `fix` - исправление ошибки;
- `docs` - документация;
- `chore` - обслуживание проекта, настройки, сборка, локальная инфраструктура;
- `test` - тесты;
- `refactor` - изменение кода без изменения поведения.

Сообщение коммита должно быть подробным. Первая строка - краткая, дальше через пустую строку идет тело с объяснением:

- что изменено;
- зачем это сделано;
- какие файлы или части проекта затронуты;
- как проверялось изменение;
- есть ли ограничения, ручные шаги или важные замечания.

Пример:

```text
docs(git): описать правила локальной работы с репозиторием

Добавлены правила работы с локальным git-репозиторием:
- зафиксирована единственная рабочая ветка master;
- запрещены команды, которые переписывают историю или удаляют коммиты;
- описан формат подробных сообщений по Conventional Commits.

AGENTS.md теперь ссылается на CONTRIBUTING.md, чтобы ИИ читал git-правила только перед выполнением git-команд.

Проверка: просмотрены итоговые файлы и состояние git.
```

## Перед коммитом

Перед созданием коммита проверь состояние репозитория:

```powershell
git status --short --branch
```

Убедись, что в коммит попадают только нужные файлы. Секреты, локальные данные, сборочные артефакты и временные каталоги не должны добавляться в репозиторий.

Если менялись файлы приложения, соблюдай локальные правила из `AGENTS.md` про сборку и деплой пакета для Yandex Cloud Functions.

## После коммита

После коммита проверь результат:

```powershell
git status --short --branch
git log -1 --format=fuller
```

В ответе пользователю укажи хеш коммита и коротко опиши, что в него вошло.
